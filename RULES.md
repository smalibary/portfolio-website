# RULES — always-on, read first

Anything that's a *decision* or a *constraint* lives here. CLAUDE.md is the
router; this file is the law. Before implementing anything new — a page,
a feature, a piece of copy, a route, a schema — re-read the relevant
section here.

If you find yourself wanting to deviate from a rule below, **don't decide
silently.** Surface it to Salem, justify it, and if the rule should change,
update *this* file as part of the same commit. Rules drift fast when they
live in chat history; that's why they're in version control.

---

## 1. Writing — academic and content

| Domain | Rule |
|---|---|
| Spelling | **Australian English** (behaviour, organisation, analyse, colour, programme, centre, recognise, specialised, utilise) |
| Citations | **APA 7th edition** — mix `(Author, Year)` parenthetical + `Author (Year)` narrative. Group at end: `(A1, Year; A2, Year)` |
| Tone | Formal, assertive, evidence-backed. Long, dense sentences with strategic hedging. Claim → evidence → elaboration → implication |
| Voice | When to use narrative citations: when the author's identity matters to the argument |

## 2. Arabic content rules

| Rule | Why |
|---|---|
| **No em dashes (—)** in Arabic. Use `،` (Arabic comma), `بعكس`, `بعد كدا`, or `و لكن` instead | Em dashes break Arabic visual rhythm and are an English-typography import |
| **Hejazi Saudi dialect**, not formal Arabic, not general Saudi | Brand voice — see `context/voice.md` |
| **Arabic primary** on every public page — RTL, larger type, presented first | Bilingual hierarchy: Arabic primary, English secondary |
| **`<bdi>` wrap** mixed Arabic + Latin digit + Latin word | Without it, `على 3 projects` renders as `projects 3` |
| **`<html lang="ar" dir="rtl">`** | Set via `Document(lang: 'ar')` and CSS |

## 3. Public-facing copy must flow from `context/`

Any text that describes Salem, the brand, or what he covers — site about
pages, LinkedIn bio, video descriptions, talk bios, book pitch, proposals
— must come from `context/`. Don't write positioning from scratch.

- He is an **"academic practitioner — architect + environmental psychology PhD"** (not "architect" alone, not "designer", not "AI educator")
- Brand covers **the built environment universe at every scale** (object → city + practice), not "AI for designers" or "creative leadership"
- Career endgame trio: **full professor / CCO / solo high-fee consultant**
- **No religion or politics framing** — see `context/guardrails.md`
- **AI is engine + one lens, not brand centre.** Don't suggest disciplining away from AI; don't make it the topic. See memory `feedback_ai_framing.md`.

If the copy you need to write would require any of the above to change,
update `context/` first (via the decisions log), then write the copy.
Don't edit positioning ad-hoc.

## 4. Content format defaults — introvert-friendly

Salem is socially nervous in performance settings. Default to text + async
formats. **Don't suggest** as a first option:
- Live podcasts, panels
- In-person speaking, conference talks
- On-camera video as a primary medium

These can be added once `context/medium-ladder.md` reaches the phase that
calls for them. Boredom with current medium = readiness signal, not
calendar dates.

## 5. SEO + AEO — required for every new public surface

When you ship a new public-facing page, route, or content type, **all of
the following must be true** before merging. Treat this as a checklist.

### 5.1 Crawlability

- [ ] Route is registered in `website-jaspr/lib/app.dart` (SSG enumerates routes statically)
- [ ] Route appears in `web/sitemap.xml` (regenerated by `tool/generate_sitemap.dart`)
- [ ] Route is **not** under `/admin/` or `/api/` if it's meant to be public (those are disallowed in `web/robots.txt`)
- [ ] Page renders meaningful HTML at build time (not "loading…" placeholder)

### 5.2 Semantic HTML

- [ ] One `<h1>` per page
- [ ] `<h2>` for major sections, `<h3>` nested where appropriate
- [ ] Use `<article>`, `<section>`, `<nav>`, `<header>`, `<footer>` semantically — not just `<div>` everywhere
- [ ] Every `<img>` has a meaningful `alt` attribute (Arabic preferred, English acceptable)

### 5.3 Structured data (JSON-LD)

- [ ] Blog posts emit `Article` schema (headline, datePublished, dateModified, image, author, mainEntityOfPage, inLanguage)
- [ ] Posts with FAQ sections emit `FAQPage` schema (auto-extracted from `## أسئلة شائعة` / `## FAQ` sections in the body)
- [ ] Research papers should emit `ScholarlyArticle` schema when #12-style detail pages are extended
- [ ] Person/Organization schema lives at the site root (`SiteData.baseUrl`)

### 5.4 Open Graph + Twitter cards

- [ ] Every blog post has an `og:image` at `/images/<post-id>/og.png` (1200×630)
- [ ] Use `tools/generate_og.py` to create cards consistent with the brand
- [ ] **#14 is not yet shipped** — when it lands, every public page must emit `og:title`, `og:description`, `og:image`, `og:type`, `twitter:card=summary_large_image`

### 5.5 Per-section discoverability (when sections become live documents — #101)

- [ ] **Tags stay article-level.** Do NOT add per-section tags. If a section is topically distinct enough to deserve its own tag page, it's a separate article (or series part via #53).
- [ ] Optional `subtopic` per section is fine *as visible metadata only* — no `/subtopic/<x>` routing
- [ ] Heading anchor IDs (#19/#20) are the right tool for sub-document URLs, not tags

### 5.6 Discoverability impact realism

- New URL = real discoverability gain (sitemap, separate index)
- New anchor (e.g. `#section-id`) = sharable fragment, modest gain
- New tag/category page = real gain (more crawlable surfaces)
- Per-section freshness signals = small SEO gain, real AEO win (AI engines cite at passage level)
- Don't overpromise SEO from features that don't add new URLs

## 6. Content data flow — never hardcode

- Public site copy lives in `website-jaspr/content/` only
  - `content/_data/site.yaml` — name, tagline, bio, hero, photos, socials, `base_url`
  - `content/_data/papers.yaml` — research papers list
  - `content/blog/<id>/post.json` + `final.md` — one folder per post
- The admin panel (`/admin/*`) writes these via `tool/save_server.dart` (`:9090`)
- Public-site components read via `lib/data/*.dart` loaders during pre-rendering
- **When adding a new piece of public-facing copy:** put it in yaml first, read via a data loader. Don't hardcode in Dart components — that breaks admin editing.

## 7. Git / GitHub — public repo discipline

- Repo is **public** at `https://github.com/smalibary/portfolio-website`. Anything committed becomes public history, even if force-pushed later.
- Root `.gitignore` excludes private working materials. **Before adding a new top-level folder with anything sensitive, update `.gitignore` first**, then commit.
- **Never track:** `context/`, `inbox/`, `mockups/`, `ROADMAP.md`, `website-jaspr/content/blog/` (drafts), `.claude/`, `.zed/`, `.dart_tool/`, `build/`, `.env*`, `*.key`, `*credentials*`
- Always tracked publicly: `website-jaspr/lib/`, `web/`, `tool/`, `pubspec.yaml`, `content/_data/site.yaml`, `content/_data/papers.yaml`, `tools/`, `CLAUDE.md`, `RULES.md`, `WORKFLOWS.md`, `website-jaspr/CLAUDE.md`, `website-jaspr/README.md`

## 8. Build + deploy

- Use `dart run tool/build.dart` for production builds, **not** `jaspr build` directly. The build script regenerates `web/sitemap.xml` first; running `jaspr build` alone leaves a stale sitemap whenever a post has been added or had its date changed.
- The save server auto-refreshes `web/sitemap.xml` after every post create/update/delete during local authoring (so dev sitemap is always fresh).

## 9. Admin panel — local-only

- Passcode `1379` is **cosmetic**, not real auth. Don't deploy `/admin/*` to production.
- The save server (`:9090`) is local-only. Never expose it.
- If admin ever needs to leave localhost, `#98` (real auth) must ship first.

## 10. Don'ts (consolidated)

- Don't hardcode site copy in Dart components — yaml first, loader second
- Don't reference files in `inbox/` from production code; move them first
- Don't change the visual design without a mockup pass (Track 1 or Track 2 — see WORKFLOWS.md)
- Don't add a third locale until ar is fully wired alongside en
- Don't deploy `/admin/*` to production — passcode is cosmetic
- Don't write public positioning text without reading `context/CLAUDE.md` first
- Don't suggest live-performance content formats before the medium ladder reaches that phase
- Don't track `context/`, `inbox/`, `mockups/`, `ROADMAP.md`, or blog drafts in git — listed in root `.gitignore` for a reason
- Don't pass children to Jaspr `script(...)` — use `content:` (inline JS) or `src:` (external URL) only
- Don't use em dashes (—) in Arabic content
- Don't add per-section tags (article-level only — see §5.5)
- Don't promise big SEO gains from features that don't add new URLs
