## 5. SEO + AEO â€” required for every new public surface

When you ship a new public-facing page, route, or content type, **all of
the following must be true** before merging. Treat this as a checklist.

### 5.1 Crawlability

- [ ] Route is registered in `website-jaspr/lib/app.dart` (SSG enumerates routes statically)
- [ ] Route appears in `web/sitemap.xml` (regenerated by `tool/generate_sitemap.dart`)
- [ ] Route is **not** under `/admin/` or `/api/` if it's meant to be public (those are disallowed in `web/robots.txt`)
- [ ] Page renders meaningful HTML at build time (not "loadingâ€¦" placeholder)

### 5.2 Semantic HTML

- [ ] One `<h1>` per page
- [ ] `<h2>` for major sections, `<h3>` nested where appropriate
- [ ] Use `<article>`, `<section>`, `<nav>`, `<header>`, `<footer>` semantically â€” not just `<div>` everywhere
- [ ] Every `<img>` has a meaningful `alt` attribute (Arabic preferred, English acceptable)

### 5.3 Structured data (JSON-LD)

- [ ] Blog posts emit `Article` schema (headline, datePublished, dateModified, image, author, mainEntityOfPage, inLanguage)
- [ ] Posts with FAQ sections emit `FAQPage` schema (auto-extracted from `## Ø£Ø³Ø¦Ù„Ø© Ø´Ø§Ø¦Ø¹Ø©` / `## FAQ` sections in the body)
- [ ] Research papers should emit `ScholarlyArticle` schema when #12-style detail pages are extended
- [ ] Person/Organization schema lives at the site root (`SiteData.baseUrl`)

### 5.4 Open Graph + Twitter cards

- [ ] Every blog post has an `og:image` at `/images/<post-id>/og.png` (1200Ã—630)
- [ ] Use `tools/generate_og.py` to create cards consistent with the brand
- [ ] **#14 is not yet shipped** â€” when it lands, every public page must emit `og:title`, `og:description`, `og:image`, `og:type`, `twitter:card=summary_large_image`

### 5.5 Per-section discoverability (when sections become live documents â€” #101)

- [ ] **Tags stay article-level.** Do NOT add per-section tags. If a section is topically distinct enough to deserve its own tag page, it's a separate article (or series part via #53).
- [ ] Optional `subtopic` per section is fine *as visible metadata only* â€” no `/subtopic/<x>` routing
- [ ] Heading anchor IDs (#19/#20) are the right tool for sub-document URLs, not tags

### 5.6 Discoverability impact realism

- New URL = real discoverability gain (sitemap, separate index)
- New anchor (e.g. `#section-id`) = sharable fragment, modest gain
- New tag/category page = real gain (more crawlable surfaces)
- Per-section freshness signals = small SEO gain, real AEO win (AI engines cite at passage level)
- Don't overpromise SEO from features that don't add new URLs

## 10. Visual work â€” browser-verify loop (mandatory)

Any visual task (CSS changes, layout tweaks, new components, design fixes)

1. Make the change
2. Use `agent-browser` to take a screenshot â€” **save screenshots to
   `inbox/`**, not the system temp directory
3. Evaluate: does it look right? Check alignment, spacing, consistency

4. If something is off â€” fix it and re-screenshot. Repeat until it looks good
5. Test **both themes** (dark + light) â€” switch with the admin rail toggle

**Don't show broken work.** Iterate silently with screenshots until it passes

no exceptions. This includes: new pages, component changes, CSS refactors,
token migrations, layout tweaks, responsive fixes, colour changes, and
any other work that affects what the user sees on screen.

**No visual change ships without a passing screenshot.**

## 11. Design â€” follow the system, minimal components

- **Every visual change must reference `DESIGN.md`** â€” use existing tokens

  components (card, pill, tag, button, input, blockquote). Differentiate with
  tokens (radius, spacing, border, accent colour), not new CSS classes or

- Before creating a new component, ask:
  1. Can an existing component with different token values do the job?
  2. Is the structural difference genuine (different HTML shape) or just
